# EIP-7702 "trade from the user's own address" rail — DELIVERED

A second, additive trading rail where trades execute with `msg.sender = the user's own EOA` (via
EIP-7702 delegation + a scoped session key). The user's funds are spent, output returns to the user,
no agent EOA holds funds, and **every money-bound is enforced on-chain** in the delegate. The existing
agent-EOA `TradeGuard` rail and the x402 rail are untouched.

## Security model held
Model decides WHAT/WHEN → on-chain cage decides ALLOWED → the user's EOA DOES. A compromised session
key or keeper cannot widen the allow-list, raise caps, redirect output (no recipient field exists —
it's pinned to the user), or beat the on-chain slippage floor. Caps are denominated in the session's
funding token (one funding token per session) to stay oracle-free.

## Two real bugs found + fixed
1. A delegated EOA couldn't receive plain ETH → added `receive()` to the delegate (a real-world
   requirement, not just test plumbing).
2. The 7702 type-4 tx must target the zero address, not the user — empty calldata to the now-delegated
   user falls into the delegate with no matching function and reverts.

## How to run locally
```bash
# 1) anvil with Prague + a high balance (the venue seeds 3M WXPL → needs lots of ETH)
anvil --balance 200000000 --hardfork prague          # see the gotcha below

# 2) deploy (writes deployments/local.json)
cd contracts && forge script script/Deploy.s.sol:Deploy --rpc-url $RPC_URL --broadcast

# 3) contract tests
forge test --match-contract AgentSessionDelegateTest

# 4) off-chain tests + scenarios (need the live chain)
pytest backend/tests/test_session.py backend/tests/test_trader_7702.py backend/tests/test_session_ctl.py
python scripts/scenario_7702.py            # Flows A–F

# 5) frontend
cd studio-frontend && pnpm build
```

## ⚠️ Local anvil / port gotcha (bit me this session)
A **Docker port-proxy was listening on `*:8545` (IPv6)** and shadowed `localhost:8545`, silently
serving a *default-balance* (10k ETH) chain — so deploys hit the wrong chain and `WXPL.deposit{value:
3M ether}` failed `OutOfFunds`, and contracts appeared "missing". Fix used: run anvil on **`--port
8546`** and point the SDK at it with `RPC_URL=http://127.0.0.1:8546` (the Python config reads `RPC_URL`
from env; the `.env` default is `localhost:8545`). If contracts look undeployed, check `lsof -nP
-iTCP:8545 -sTCP:LISTEN` for a `com.docker` listener before assuming the chain is broken.

## Plasma testnet 7702 — verification checklist
EIP-7702 support on Plasma is the gating unknown. `scripts/probe_7702_support.py` answers it with the
minimal proof (a type-4 SetCode tx + a designator/view check) — no venue needed. Run it FIRST.

1. **Probe support** (cheap, one funded key):
   ```bash
   RPC_URL=$PLASMA_RPC_URL PROBE_PK=$FUNDED_TESTNET_KEY DELEGATE_ADDR=0x<delegate> \
     python scripts/probe_7702_support.py     # exits non-zero if NOT supported
   ```
   (If no delegate is deployed yet, deploy just `AgentSessionDelegate` first, or point `DELEGATE_ADDR`
   at any contract — the probe only needs a code-bearing target.)
2. **Deploy the venue + delegate** (needs a funded `RELAYER_PK` + native for the WXPL seed; lower it on
   a constrained faucet):
   ```bash
   WXPL_SEED_WEI=<fits your faucet> RELAYER_PK=$PLASMA_RELAYER_PK \
     forge script contracts/script/Deploy.s.sol:Deploy --rpc-url $PLASMA_RPC_URL --broadcast
   ```
   The deploy preflights balance + chainId and fails loud if underfunded / wrong chain.
3. **Sanity the deployment from Python**: `LocalAdapter().assert_chain_ready(require_session=True)`.
4. **Run the full rail** against testnet: point `RPC_URL`/`CHAIN_ID` at Plasma and run
   `python scripts/scenario_7702.py` (Flows A–F). Note: the demo wallet keys in
   `session_ctl.DEMO_ANVIL_KEYS` are anvil-only — for testnet, drive with real funded keys.
5. **Blocked here because**: no funded Plasma `RELAYER_PK` / testnet RPC in this environment. Everything
   above is ready to run once those exist.
